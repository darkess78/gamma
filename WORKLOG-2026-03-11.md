## Additional overnight progress
- Added `gamma/voice/controller.py`, a structure-first voice mode controller that wraps the existing STT -> conversation -> TTS path instead of burying behavior in a single mic script.
- Defined mode policies for:
  - `turn-based`
  - `streaming`
  - `always-listening`
  - `interruptible`
- Added `python -m gamma.run_voice_mode` as the new controller-oriented CLI entry point.
- Kept the old `python -m gamma.run_mic_voice_loop` entry point alive by routing it through the controller.
- Chose an explicit fallback strategy: non-turn modes are modeled now, but `streaming` and `interruptible` fall back to the validated turn-based runtime path until real duplex audio / VAD / partial transcript plumbing exists.

## Blockers / caveats
- GPT-SoVITS support is only a first contract, not a validated integration yet. Different forks expose different payload shapes; this adapter assumes a simple JSON POST that returns WAV bytes.
- The controller design is in better shape now, but real always-listening / streaming behavior still needs hardware validation and likely a different capture backend than fixed-length `arecord`.
- True barge-in is not implemented yet; the mode policy is defined, but playback interruption and continuous listening are still future work.

## Recommended next move
1. Run the mic loop on real hardware and capture the first friction points.
2. Point the TTS layer at an actual GPT-SoVITS server and adjust the request/response contract based on the real fork in use.
3. If the mic path behaves, add lightweight turn-end detection / VAD instead of jumping straight to fully streaming audio.
